Data Models
OpsKnowledgeHQ
Purpose:
A centralized, read-only knowledge store for operational data including sites, phones, nodes, troubleshooting procedures, references, and operator notes. This table must support comprehensive filtering and search functionality.

Fields:

Common Fields:

Title: General description or name of the record.
RecordType: Choice field indicating the type (e.g., Site, Phone, Node).
Tags: Metadata for easy categorization and filtering.
Summary: A one-line description of the record’s purpose or status.
Details: Comprehensive notes or long-form description.
Links: Stores associated URLs for reference documents or tools.
OwnerTeam: The team responsible for the record.
IsPinned: Boolean to mark records as high-priority or frequently accessed.
IsActive: Boolean to indicate whether the record is active.
LastVerified: Date-time field for the last manual verification.
UpdatedByDisplay: User or team who last updated the record.
Site-Specific Fields:

SiteCode: Unique identifier for the site.
SiteName: Name of the site.
Address: Physical address of the site.
City, State, Zip: Geographical location details.
CircuitId: Identifier for the site’s network circuit.
SiteAffiliate: Associated organization or operating affiliate.
SiteNotes: Additional notes specific to the site.
Phone-Specific Fields:

ContactName: Name of the point of contact at the site.
Department: Department name of the contact.
Role: Contact’s role (e.g., manager, technician).
IvantiPhonePrimary, IvantiPhoneSecondary: Primary and secondary contact phone numbers derived from Ivanti records.
StaffDirectoryPhone: Phone number from company directory.
OtherContactLabel, OtherContactPhone: Custom label and number for additional contacts.
Email: Contact’s email address.
AfterHoursNotes: Specific instructions or escalation procedures for after-hours access.
Node-Specific Fields:

AppGroup: Associated application or service group.
NodeHostname: Hostname of the node.
NodeIPAddress: IP address of the node.
NodeRole: Node’s operational role (e.g., database, web server).
NodeEnvironment: Node environment (e.g., production, staging).
NodeOwnerTeam: Team responsible for the node.
OpsOncallExports
Purpose:
Tracks on-call schedules and roles for triage operations with advanced functionality to replace all on-call records using Excel import.

Fields:

OnCallRole: Designated role for the on-call personnel (e.g., Network Administrator).
PrimaryName, PrimaryPhone: Name and phone number of the primary operator.
SecondaryName, SecondaryPhone: Backup operator details.
CoverageNotes: Specialized notes about on-call duties or coverage.
SharedNotes: General shared notes for handoffs or escalations.
Tags: Metadata for filtering purposes.
IsActive: Boolean to indicate whether the on-call role is currently active.
LastVerified: Date-time field for the last manual verification of the record.
UpdatedByDisplay: User/team who last updated the record.
Constraints:

Records are read-only with optional manual refresh for updates.
Update Button must enable bulk replacement of all on-call data via Excel import.
Import Excel Fields:

Primary, Primary Phone, Backup, Backup Phone, Manager, Director, VP, StartDate, EndDate.
Features:

Ensures administrators can update schedules in bulk using a structured Excel file.
Prevents manual entry errors and allows streamlined workflow for on-call data management.
OpsCommands
Purpose:
Provides a categorized repository of command-line scripts for manual operator usage, supporting rapid troubleshooting and diagnostics.

Fields:

ScriptName: Descriptive name/title of the script.
ScriptText: Full content of the CMD/PowerShell script.
ShellType: Validates the script’s type (CMD or PowerShell).
Category: Defines the functional purpose of the script (e.g., network diagnostics, user management).
Tags: Metadata for enhanced search and filtering.
Summary: A brief explanation of the script’s use case.
ModifiedDate: Date-time when the script was last modified.
UpdatedByDisplay: User/team who last updated the script.
Constraints:

Scripts are copy-to-clipboard only. They are viewable and cannot be executed in-app.
Organized categorically for easy access and structured use.
Features:

Built-in copy-to-clipboard functionality for operators to quickly share and execute scripts externally.
Categorization ensures operators find relevant scripts efficiently.
